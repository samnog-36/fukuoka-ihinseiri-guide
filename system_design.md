# 福岡遺品整理ガイド - 業者マッチング・ポータル化 全体設計図

本設計図は、現在作成中の静的SEOブログサイト（フェーズ1）をベースに、将来的に**業者登録・マッチング・掲載料徴収**の機能を持たせたポータルサイト（フェーズ2）へ拡張するための完全な仕様書です。

---

## 1. システムアーキテクチャ方針

静的HTMLのままでは業者登録やデータベース保存ができないため、以下の構成でシステムを構築します。

*   **フロントエンド（見た目）**: 現在のHTML/CSSデザインをそのままReactコンポーネント化して引き継ぐ（SEO・表示速度を維持）
*   **バックエンド（裏側）**: Manusの `web-db-user` テンプレートを利用
    *   Node.js (APIサーバー)
    *   MySQL / TiDB (データベース)
    *   Drizzle ORM (データベース操作)
    *   Manus-Oauth (業者ログイン認証)
*   **インフラ・ホスティング**: Manusの自動デプロイ環境を利用（サーバー管理不要）

この構成により、SEOに強いブログメディアとしての強みを維持しつつ、高度なポータル機能を実装できます。

## 2. データベース設計（主要テーブル）

業者が登録する情報と、ユーザーからの問い合わせを管理するためのデータベース構造です。

### 2.1. 業者テーブル (`vendors`)

業者が登録する基本情報と対応可能なジャンルを保存します。

| カラム名 | データ型 | 必須 | 説明 |
| :--- | :--- | :--- | :--- |
| `id` | UUID | ◯ | 業者の一意のID |
| `company_name` | VARCHAR | ◯ | 会社名・屋号 |
| `representative` | VARCHAR | ◯ | 代表者名 |
| `phone` | VARCHAR | ◯ | 電話番号 |
| `email` | VARCHAR | ◯ | メールアドレス（ログインID兼通知先） |
| `address` | VARCHAR | ◯ | 所在地（福岡県内の住所） |
| `website_url` | VARCHAR | | 自社サイトURL |
| `license_info` | TEXT | ◯ | 許認可情報（古物商許可番号、一般廃棄物収集運搬許可など） |
| `is_published` | BOOLEAN | ◯ | 掲載状態（デフォルト: false。掲載料入金確認後に運営がtrueにする） |
| `created_at` | TIMESTAMP | ◯ | 登録日時 |

### 2.2. 対応ジャンルテーブル (`vendor_categories`)

業者が「何を取り扱えるか」を複数選択できるようにする中間テーブルです。

*   **対応ジャンルマスタ**:
    1. 遺品整理
    2. 生前整理
    3. 特殊清掃
    4. ゴミ屋敷清掃
    5. 不用品回収・買取
    6. 遺品供養
    7. ハウスクリーニング

| カラム名 | データ型 | 必須 | 説明 |
| :--- | :--- | :--- | :--- |
| `vendor_id` | UUID | ◯ | 業者ID |
| `category_id` | INT | ◯ | 対応ジャンルID（1〜7） |

### 2.3. 対応エリアテーブル (`vendor_areas`)

業者が「福岡県内のどの地域に対応可能か」を複数選択できるようにします。（例：福岡市全域、北九州市のみ、県内全域など）

### 2.4. 問い合わせテーブル (`inquiries`)

一般ユーザーからの相談内容を保存します。

| カラム名 | データ型 | 必須 | 説明 |
| :--- | :--- | :--- | :--- |
| `id` | UUID | ◯ | 問い合わせID |
| `user_name` | VARCHAR | ◯ | 相談者名 |
| `user_phone` | VARCHAR | ◯ | 連絡先電話番号 |
| `user_email` | VARCHAR | ◯ | 連絡先メールアドレス |
| `target_area` | VARCHAR | ◯ | 作業希望エリア（例：福岡市博多区） |
| `required_service` | VARCHAR | ◯ | 希望サービス（遺品整理、特殊清掃など） |
| `property_type` | VARCHAR | ◯ | 間取り・物件種別（1K、3LDKなど） |
| `message` | TEXT | ◯ | 相談内容詳細 |
| `status` | VARCHAR | ◯ | ステータス（未対応、対応中、成約済など） |
| `created_at` | TIMESTAMP | ◯ | 問い合わせ日時 |

## 3. 機能要件

### 3.1. 業者向け機能

#### 1. 業者登録フォーム
*   **入力項目**: 会社名、代表者名、電話、メール、住所、URL、許認可情報
*   **チェックボックス**: 対応可能ジャンル（複数選択可）、対応可能エリア（複数選択可）
*   **フロー**: フォーム送信 → データベースに「未掲載」状態で保存 → 運営（あなた）に通知メール送信。

#### 2. 業者ダッシュボード（マイページ）
*   **ログイン**: メールアドレスとパスワード（またはOAuth）でログイン。
*   **機能**: 
    *   自社情報の編集・更新。
    *   自社宛に届いた問い合わせ（案件）の閲覧。

### 3.2. 運営（あなた）向け機能

#### 1. 広告掲載料・入金管理フロー
*   **決済方法**: 銀行振込（初期フェーズ）
*   **料金設定（案）**:
    *   **月額掲載プラン**: 月額 10,000円（サイト上の業者一覧に掲載される）
    *   **成果報酬プラン**: 問い合わせ1件につき 3,000円（または成約時10%等。最初は管理が楽な月額固定を推奨）
*   **運用フロー**:
    1. 業者が登録フォームから申請。
    2. 運営（あなた）が内容を審査し、問題なければ「振込先口座情報」をメールで案内。
    3. 業者が銀行振込を実施。
    4. 運営が入金を確認後、管理画面（またはDB直接操作）で該当業者の `is_published` を `true`（掲載中）に変更。
    5. サイトの「おすすめ業者一覧」や「地域別業者一覧」に自動的に表示されるようになる。

#### 2. 問い合わせ自動通知機能
*   一般ユーザーがサイトの「無料相談フォーム」から問い合わせを送信した際の挙動：
    1. データベース（`inquiries`）に保存。
    2. **運営（あなた）への通知**: 「新規の問い合わせがありました」というメールが届く。
    3. **業者への通知（マッチング）**: 問い合わせの「希望エリア」と「希望サービス」に合致する、かつ「掲載中（`is_published`=true）」の業者に対して、一斉に通知メールを自動送信する。
        *   *※初期は運営が手動で業者に案件を振る（メール転送する）「コンシェルジュ方式」も品質担保の観点でおすすめです。*

### 3.3. 一般ユーザー向け機能

*   **SEOブログ・ガイド記事**: 現在作成している静的HTMLベースの高品質な記事群。
*   **業者検索・一覧表示**: データベースに登録され、掲載中となっている業者を、地域別やジャンル別に絞り込んで表示する機能。
*   **無料相談・見積もりフォーム**: ユーザーが要件を入力して送信するフォーム。

## 4. 開発・納品フェーズの進め方

本システムは、ユーザー様からの「GO」サインが出た後、**全ての機能を完全に実装し、動作確認を終えた状態で納品**します。「後で作る」部分は一切残しません。

### 開発ステップ

1.  **プロジェクト初期化**: Manusの `web-db-user` スキャフォールドを用いて、React + Node.js + DB環境を構築。
2.  **DB構築**: 上記設計に基づくテーブルの作成。
3.  **フロントエンド移植**: 現在作成済みの静的HTML（SEOブログ部分）をReactコンポーネントに変換し、組み込み。
4.  **業者機能実装**: 登録フォーム、ダッシュボードの実装。
5.  **メール通知実装**: 問い合わせ時の自動メール通知システムの実装（SendGrid等のメール配信サービス連携）。
6.  **テスト**: 業者登録〜入金確認（フラグ変更）〜問い合わせ〜業者への通知、の一連のフローをテスト。
7.  **納品**: 完全に動作するシステムのURLおよびソースコード一式を納品。

## 5. 広告掲載料に関する提案

最初は「銀行振込」での対応とするため、運営の手間を減らしつつ収益化しやすいモデルとして以下を推奨します。

**【推奨プラン：月額固定掲載料モデル】**
*   **金額**: 月額 10,000円 〜 15,000円（税別）
*   **理由**:
    *   成果報酬型（問い合わせ課金や成約課金）は、「本当に成約したか」を業者が隠すリスクがあり、確認の手間が膨大になります。
    *   月額固定であれば、「毎月◯日に振り込まれているか」を確認し、振り込まれていなければサイトから非表示にする（`is_published` を false にする）だけで管理が完結します。
    *   サイトのアクセス数（SEO順位）が上がれば、「月額3万円」などに値上げしていくことが可能です。

以上が、業者マッチング機能を含む全体設計図です。
